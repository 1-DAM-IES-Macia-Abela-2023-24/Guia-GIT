  1. ¿Qué es GIT?

    2. GIT es un sistema de control de versiones desarrollado (VCS del inglés Version Control System) por Linus Torvalds. Se creo inicialmente para el desarrollo de forma colaborativa de proyectos de gran envergadura como el core de Linux.

    La idea de Linus era disponer de un sistema de control de versiones distribuido para que en caso de pérdida del repositorio, digamos central, se pudiera recuperar fácilmente a partir de las copias del resto de repositorios.

  2. ¿Cómo funciona GIT?

    3. Para ello, Linus desarrollo un VCS distribuido no incremental. Con esto último, queremos decir que en el repositorio no se guardan archivos con el incremental respecto a la versión anterior, sino que se guardan copias completas del archivo y si no ha sido modificado, se apunta hacia la última versión en la que el archivo fue modificado, como puede verse en la imagen siguiente:

    Por dentro, GIT dispone de 3 áreas diferenciadas de trabajo:

    • Directorio de trabajo

      • Carpeta del sistema donde dejo los archivos del código fuente

    • Staging

      • Zona interna de GIT donde prepara los cambios para ser almacenados en el repositorio. Se le suele llamar índice pues es donde se almacenan todos los cambios hasta el próximo comit. Se puede dejar la modificación de varios ficheros en staging hasta haber completado la funcionalidad en desarrollo.

    • Repositorio (.git)

      • Es donde mantenemos el histórico de versiones de nuestro producto para así poder recuperar fácilmente una versión determinada en caso de hacernos falta.

  3. Estados de los archivos en GIT

    4. Para saber como tenemos nuestro repositorio, en GIT se guarda el estado en el que se encuentran cada uno de nuestros archivos tanto en el área del directorio de trabajo como en el área de Staging. Principalmente tenemos 4 estados:

    • Untracked

      • Los archivos marcados como Untracked son archivos que no se encuentran en monitoriozación en el repositorio. Es decir, no están incluidos en el repositorio. En la versión reducida del comando de estado (git status) aparecerán con un signo de interrogación en rojo (?)

    • Unmodified

      • En este estado tendremos los archivos que son idénticos a lo que GIT tiene almacenado en su repositorio y por tanto no hay que hacer nada con ellos. De echo, si lanzamos un comando de estado (git status) estos ficheros no aparecerán a pesar de tenerlos en el directorio de trabajo.

    • Modified

      • En este estado veremos aquellos archivos que han sufrido una modificación en el directorio de trabajo respecto a aquellos que tiene GIT guardados en el repositorio. Se marcarán con una M tras ejecutar un comando de estado abreviado y aparecerá en la columna de la derecha.

    • Staged

      • En staged estarán los archivos que están preparados para ser almacenados en el repositorio, por tanto, son archivos con diferencias entre el directorio de trabajo/staging y el repositorio. Se marcarán con una M tras ejecutar un comando de estado abreviado y aparecerá en la columna de la izquierda.
